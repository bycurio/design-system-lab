Code Connect — How It Works

What a .figma.tsx file looks like
import figma from '@figma/code-connect' import { Button } from './Button' // Links this Figma node → Button.tsx figma.connect(Button, 'https://figma.com/design/ITyTagL...', { props: { variant: figma.enum('Variant', { Primary: 'primary', Secondary: 'secondary', Ghost: 'ghost', Danger: 'danger', }), size: figma.enum('Size', { Small: 'sm', Medium: 'md', Large: 'lg' }), label: figma.string('Label'), disabled: figma.boolean('Disabled'), }, example: ({ variant, size, label, disabled }) => ( <Button variant={variant} size={size} disabled={disabled}> {label} </Button> ), })
Publish workflow
1
Figma component properties match code props

Variant names in Figma ("Primary", "Ghost") map to TypeScript values ("primary", "ghost") via figma.enum()

2
One .figma.tsx per component in packages/ui

Co-located next to the component source — Button.tsx, Button.figma.tsx

3
figma connect publish pushes all mappings

Run manually or as a CI step. Designers immediately see real code in Dev Mode with no generated snippets.

4
Dev Mode experience

Designer selects a Button → Dev Mode shows the exact <Button variant="primary"> snippet with correct props. Copy-paste ready.

Button
Triggers an action or event. Use the primary variant for the most important action on a page.
import { Button } from '@ds/ui' <Button variant="primary">Label</Button>
The handover loop: Designer picks a component in Figma → Dev Mode shows the real <Button> snippet → developer opens docs site → sees live preview, full prop table, and token reference. No ambiguity, no translation layer. Both surfaces stay in sync because they reference the same packages/ui source.
