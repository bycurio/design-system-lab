Token Architecture

Which approach should be the source of truth for your design tokens?

A

Figma Variables → Style Dictionary → Tailwind v4

Figma is source of truth. Variables export via Tokens Studio, Style Dictionary v4 transforms to CSS vars + Tailwind config.

RECOMMENDED
Figma Variables
↓ Tokens Studio
tokens.json (W3C DTCG)
↓ Style Dictionary v4
CSS vars + tailwind.config
  • Designers own tokens in Figma
  • Bidirectional sync possible
  • Industry standard pipeline
  • Tokens Studio plugin required
  • More moving parts
Token tiers
■ Primitive --color-blue-500: #3b82f6 ■ Semantic --color-brand: var(--color-blue-500) ■ Component --button-bg: var(--color-brand)
B

Tailwind v4 CSS-first → Figma mirrors it

Code is source of truth. Define everything in Tailwind v4's @theme CSS config, then sync values back into Figma Variables.

tailwind.css @theme {}
↓ build step
CSS custom properties
↓ script/plugin
Figma Variables (sync)
  • No extra tooling (native Tailwind v4)
  • Devs are source of truth
  • Tailwind utilities auto-generated
  • Designers depend on devs for token changes
  • Figma sync is one-directional
Example
@theme {
  --color-brand: oklch(65% .25 250);
  --color-brand-hover: oklch(60% .28 250);
  --spacing-sm: 0.5rem;
}
C

W3C DTCG JSON → Style Dictionary → All outputs

Token JSON files are the source of truth (W3C Design Token Community Group spec). Style Dictionary v4 generates CSS, Tailwind, Figma, iOS, Android from one source.

tokens/*.json (DTCG)
↓ Style Dictionary v4
CSS vars / Tailwind config
Figma Variables (import)
Swift / Compose (future)
  • Most portable & future-proof
  • Platform-agnostic source
  • Multi-platform output
  • Most setup/tooling overhead
  • Figma sync still needs plugin
  • Overkill if web-only
Example DTCG token
{
 "color": {
  "brand": {
   "$type": "color",
   "$value": "#3b82f6"
  }
 }
}
